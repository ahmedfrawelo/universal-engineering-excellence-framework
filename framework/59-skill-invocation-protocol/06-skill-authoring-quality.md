# Skill Authoring Quality

## Purpose

New skills should be small, enforceable, and testable. A skill is not a dumping ground for general advice.

## Required Shape

- Clear trigger: when the skill must be used and when it must not.
- Concrete workflow: ordered steps with decision points.
- Evidence requirements: what proves the skill was applied.
- Anti-patterns: shortcut behavior the skill prevents.
- References: source docs, license, and attribution where external material influenced the skill.
- Validation: at least one test, checklist, or example proving the skill can be followed.

## Quality Rules

- Prefer one focused skill over a broad vague skill.
- Avoid copying long external text. Extract principles, cite the source, and write UEEF-native instructions.
- Do not conflict with UEEF browser, security, file-ownership, or delivery-continuation rules.
- Skills that affect UI must cooperate with `ui-ux-pro-max`, `impeccable`, and relevant design packs.

## Completion Criteria

A new skill is complete only when it is installed or documented in the expected skills path, discoverable by the runtime, and validated by a representative task.
